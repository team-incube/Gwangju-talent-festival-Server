# /planning 스킬

복잡한 기능 구현 전에 계획을 수립하고 검토한다.

## 사용법

```
/planning {구현할 기능 설명}

예시:
/planning 실시간 공연팀 랭킹 업데이트 기능 (SSE + Redis 캐시)
/planning 어드민 대시보드용 통계 API 추가
```

## 계획 수립 절차

### 1단계: 요구사항 분석
- 이 기능이 해결하는 문제는 무엇인가?
- 어떤 사용자 역할이 이 기능을 사용하는가?
- 어떤 데이터가 필요한가?

### 2단계: 현재 코드 파악
- 관련 도메인 코드 읽기
- 유사한 기능이 이미 있는지 확인
- 재사용 가능한 컴포넌트 파악

### 3단계: 기술적 접근 결정
각 결정 사항에 대해 선택지와 이유를 제시:
- 새 UseCase vs 기존 UseCase 확장
- SSE 필요 여부
- Redis 캐시 필요 여부
- 새 도메인 필요 여부

### 4단계: 구현 계획 작성

```markdown
## 구현 계획: {기능명}

### 영향받는 파일
- 신규: ...
- 수정: ...

### 구현 순서
1. 
2. 
3. 

### API 설계
| Method | Endpoint | Auth | 설명 |
|--------|----------|------|------|
| | | | |

### 리스크
- 

### 예상 작업량
- 소: 1-2개 파일
- 중: 3-5개 파일
- 대: 6개+ 파일, 새 도메인
```

### 5단계: 계획 확정 후

계획을 제시하고 승인을 받은 후 `feature` 에이전트를 호출한다.

> **Flywheel 관점:** 계획 단계에서 `.claude/flywheel/learnings.md`와
> `.claude/flywheel/patterns.md`를 먼저 읽어 과거의 실수를 반복하지 않는다.
